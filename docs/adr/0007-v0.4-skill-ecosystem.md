# ADR 0007 – v0.4 Skill Ecosystem

**Status:** Accepted  
**Date:** 2026-05-05  
**Deciders:** @alicken-lai

---

## Context

v0.3 ships relevance-ranked memory injection and a set of memory-management CLI
commands.  The skill system itself still has no machine-verifiable contract
between a skill author and the runtime: a `skill.yaml` declares `input_type`
and `output_type` as free-form strings but does not constrain *what* data those
types carry.

Three problems emerge as the skill library grows:

1. **Silent mismatches** – a runner can feed JSON data that violates the
   expected shape; the LLM receives malformed context and produces wrong output
   with no visible error.

2. **Discoverability** – there is no CLI command to list all available runnable
   skills.  Users must browse the file system to see what skills exist.

3. **Packaging** – sharing a skill today means copying a directory by hand.
   There is no formal spec for what a skill package must contain.

v0.4 addresses problems 1 and 2 (and lays groundwork for 3 and a future
community registry).

---

## Decision

### 1. JSON Schema validation for `input_schema` / `output_schema`

Skill authors may add optional `input_schema` and `output_schema` fields to
their `skill.yaml`.  Each field holds an inline JSON Schema (Draft 7) object.

```yaml
# skill.yaml example
name: meeting_minutes_to_report
input_schema:
  type: object
  required: [attendees, date, notes]
  properties:
    attendees: { type: array, items: { type: string } }
    date:       { type: string, format: date }
    notes:      { type: string }
output_schema:
  type: object
  required: [summary]
  properties:
    summary: { type: string }
```

A new module `singleclaw/skills/validator.py` introduces `SkillValidator`:

```
SkillValidator
    .validate_manifest(skill: Skill) -> ValidationResult
    .validate_input(skill: Skill, data: dict) -> ValidationResult
    .validate_output(skill: Skill, data: dict) -> ValidationResult
```

`ValidationResult` is a lightweight dataclass with:
- `is_valid: bool`
- `errors: list[str]`

**Rationale for JSON Schema Draft 7:**
`jsonschema` is already a runtime dependency (added in v0.1).  No new packages
are required.  Draft 7 is the most widely supported version and covers all
common validation constraints.

**Schemas are optional.**  Skills without `input_schema` / `output_schema` pass
validation with a warning (not an error), preserving backward compatibility.

### 2. `singleclaw skills list` CLI command

A new `skills_app` Typer sub-application (name: `skills`, plural – distinct
from the existing `skill show` for guidance skills) exposes:

```
singleclaw skills list
```

Renders a Rich table showing every discovered runnable skill:

| Column | Source |
|--------|--------|
| Name | `skill.yaml` → `name` |
| Version | `skill.yaml` → `version` |
| Description | `skill.yaml` → `description` (truncated at 60 chars) |
| Risk | `skill.yaml` → `risk_level` |
| Schemas | ✓ / – for `input_schema` and `output_schema` presence |
| Status | ✅ valid / ❌ invalid |

### 3. `singleclaw skills validate <skill_name>` CLI command

```
singleclaw skills validate <skill_name>
```

1. Looks up the skill in `SkillRegistry`.
2. Runs `SkillValidator.validate_manifest(skill)`.
3. Prints a Rich report: green ✔ for each passing check, red ✘ with message
   for each failing check.
4. Exits with code 0 on success, code 1 on any validation error.

### 4. Skill packaging spec (deferred to v0.4 sub-issue)

The packaging spec (`docs/skill-packaging.md`) and `pyproject.toml` entry-point
convention are documented separately.  v0.4 implementation focuses on issues A,
B, and C.  Issues D and E (packaging + registry) are deferred to follow-on PRs.

### 5. Community registry index format (deferred)

The community registry design is intentionally out of scope for this ADR.  A
future ADR will define the local index YAML schema and remote registry protocol.
The `singleclaw skills list` command is designed to later accept a `--registry`
flag without breaking the current interface.

---

## Consequences

**Positive:**
- Skill authors can catch data-contract violations before LLM calls are made.
- `SkillValidator` is independently testable; its internals can later be swapped
  for a stricter schema version without changing the CLI.
- The `skills list` command gives users immediate discoverability from the
  terminal.

**Negative / Trade-offs:**
- `input_schema` and `output_schema` are optional for now.  A skill that omits
  them passes validation silently, so the guarantee is weaker than a fully
  mandatory schema.  This is intentional for backward compatibility; a future
  ADR may enforce schemas on new skills.
- Validating the schema definition itself (meta-validation) adds a small
  complexity cost, but `jsonschema.Draft7Validator.check_schema()` handles this
  at negligible overhead.

**Architecture invariants preserved:**
- `singleclaw/skills/runner.py` does not import `SkillValidator`; validation is
  opt-in and orchestrated by the CLI layer.
- All destructive actions still pass through `GuardianPolicy.check()`.
