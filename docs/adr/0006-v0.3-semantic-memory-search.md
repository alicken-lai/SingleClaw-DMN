# ADR 0006 – v0.3 Semantic Memory Search

**Status:** Accepted  
**Date:** 2026-05-05  
**Deciders:** @alicken-lai

---

## Context

v0.2 injects memory into skill prompts by taking the **5 most recent** DMN
records (`MemoryStore.recent(n=5)`).  This is a useful starting point but has
two problems:

1. **Relevance** – recent items are often unrelated to the current task.
   A skill about a procurement decision should see past procurement notes, not
   last night's personal reminder.

2. **Scale** – as the memory store grows beyond a few hundred records, passing
   the most recent 5 still misses older but highly relevant context.

v0.3 will replace the "most recent N" heuristic with **relevance-ranked
retrieval**, so `SkillRunner` always receives memory items that actually help.

---

## Decision

### Search approach: keyword TF-IDF cosine similarity (first iteration)

Use a **pure-Python keyword cosine similarity** approach for v0.3.  Full neural
embeddings (sentence-transformers, OpenAI embeddings API) are deferred to a
future version.

Rationale:

| Approach | Pros | Cons | Decision |
|----------|------|------|----------|
| Recent-N (v0.2 status quo) | Zero overhead | Low relevance | ✗ replace |
| TF-IDF cosine | No new deps, fast, works offline | Not semantic | ✓ **v0.3** |
| sentence-transformers | True semantic similarity | Adds ~500 MB model download | Deferred |
| LLM embedding API | Best quality | Requires network + API cost per query | Deferred |

TF-IDF is implemented using Python's standard library (`collections.Counter`).
No new runtime dependencies are required for the initial implementation.

### New module: `singleclaw/dmn/search.py`

Introduces `MemorySearch` – a lightweight retrieval class:

```
MemorySearch(store: MemoryStore)
    .query(text: str, top_k: int = 5) -> list[dict]
```

- Builds a TF-IDF index over the `content` field of all memory records at
  query time (no pre-build step required).
- Returns the `top_k` records ranked by cosine similarity to `text`.
- Falls back to `MemoryStore.recent(n=top_k)` when the store is empty or
  all scores are zero.

### CLI changes

Two new sub-commands under the `memory` group:

| Command | Description |
|---------|-------------|
| `singleclaw memory list [--tag TAG]` | List all memory items; filter by tag |
| `singleclaw memory search "query"` | Search memory by relevance and print results |

Memory items may carry an optional `tags` field (list of strings) written at
`remember` time (future: `singleclaw remember --tag meeting "…"`).  For now,
`memory list --tag` filters by exact string match in the existing `tags` field
if present.

### Context injection update

`singleclaw/cli.py` – `run` command:

- Replace `MemoryStore.recent(n=5)` with
  `MemorySearch(store).query(skill.description or input_text, top_k=5)`.
- Skill description is used as the query when no input file is provided;
  otherwise, the raw input text is used.

### Memory export

`singleclaw memory export [--format markdown|json] [--output PATH]`

Exports the full memory store to a single file.  Default format: Markdown
(human-readable); JSON for machine consumption.  Implemented as a read-only
operation; Guardian `ALLOW` classification.

### Memory pruning / archiving

`singleclaw memory archive --before DATE`

Moves records older than `DATE` to `.singleclaw/memory_archive.jsonl`,
preserving the active store for fast reads.  This is a **destructive action**
(records are removed from the live file) and **must pass `GuardianPolicy.check()`**
before execution.

---

## Consequences

**Positive:**
- Relevance-ranked context injection improves skill output quality with no
  extra cost or latency.
- `MemorySearch` is independently testable and swappable.
- Adding neural embeddings later only requires replacing `MemorySearch.query()`
  internals; the interface and CLI are unchanged.

**Negative / Trade-offs:**
- TF-IDF cannot handle paraphrase or synonym matching ("vendor" vs "supplier").
  This is accepted for v0.3; neural embeddings address it in a later version.
- Rebuilding the TF-IDF index on every `query()` call is O(n) in memory size.
  For stores up to ~10,000 records this is imperceptible; larger stores will
  need a caching layer.

**Architecture invariants preserved:**
- `singleclaw/skills/runner.py` still does not import `singleclaw/dmn/`.
  The CLI orchestrator (`cli.py`) calls `MemorySearch` and passes results as a
  plain `list[dict]`.
- All destructive memory operations go through `GuardianPolicy.check()`.
